# Security Consistency Audit — 2026-06-09

Successor to the 2026-05-28 endpoint security audit
(`endpoint-security-audit-2026-05-28.md`, remediated via issues #431–#442 and
ADR-400). That round fixed *what the endpoints enforce*. This round audited
*whether anything enforces the conventions* — and found that the platform's
security posture rests almost entirely on discipline, not machinery.

Five surfaces were swept in parallel (read-only): endpoint auth consistency,
Cypher injection surface, infrastructure/secrets exposure, data-model/response
hygiene, and existing deterministic enforcement. Load-bearing claims below were
individually re-verified against the working tree at commit `7a1bea7a5`.

## Headline

The dominant patterns are good: `$param` binding or regex-validated
interpolation for Cypher, uniform `require_permission` dependencies, secrets
generated by `operator.sh init` and never baked into images, `0600` CLI token
files, password hashes isolated in `*InDB` models. The gap is enforcement:

- The only security linter wired to CI runs with `|| true`
  (`.github/workflows/lint.yml:25`) — failures never block.
- ~2,000 lines of auth/permission tests exist (`tests/api/test_endpoint_security.py`,
  `test_job_permissions.py`, eight `test_*_auth.py` files) but never run in CI.
- No secret scanning, dependency auditing, or SAST anywhere in the pipeline.

Evidence that discipline alone drifts: issue #439 (May cluster) replaced
`get_current_user` with `get_current_active_user` on artifacts/programs/grants
reads, yet `api/app/routes/query_definitions.py:37,116` still uses
`get_current_user` today — disabled accounts retain access to those endpoints.

## Verified findings

### F1 — Dormant Cypher injection vectors (verified)

Two methods interpolate caller strings into quoted Cypher with f-strings,
bypassing the codebase's own `_validate_*` pattern:

- `api/app/lib/query_facade.py:375-376` — `category` in
  `find_vocabulary_synonyms()`
- `api/app/lib/age_client/query.py:404,408` — `creator`, `cognitive_leap` in
  `list_learned_knowledge()`

Neither is currently reachable from a route (dormant, not exploitable), but
nothing prevents a future route from wiring them. The safe pattern they should
follow lives in `graph_facade.py` (`_validate_rel_types`,
`_validate_concept_ids`).

### F2 — Error-detail leakage is systematic (verified count)

155 occurrences of `str(e)` in `api/app/routes/`, predominantly
`HTTPException(detail=str(e))` or f-string equivalents. Driver/AGE exception
text can carry table names, query fragments, and connection details to
clients. The correct pattern (generic detail + `logger.error(...,
exc_info=True)`) already exists in the codebase alongside the leaky one.

### F3 — Response-model coverage is bimodal

~68% of 226 endpoints declare `response_model=`. Five modules sit at 0%
(`admin_workers.py`, `models.py`, `sources.py`, `storage_admin.py`,
`vision.py`); `admin.py` is at 2/14. Undeclared models mean no schema firewall
against future internal-field leakage.

### F4 — Auth is uniform except where hand-rolled

- `admin_workers.py:39-43` — private `_check_permission()` helper instead of
  `Depends(require_permission(...))`.
- `jobs.py` — bespoke `JobPermissionContext` (defensible ownership semantics,
  but a second authz model to audit).
- `query_definitions.py:37,116` — `get_current_user` instead of
  `get_current_active_user` (verified; see headline note re #439).

### F5 — Infra: dev/prod divergence, missing headers, weak example secrets (verified)

- `docker/docker-compose.yml:38,92-93` publishes Postgres 5432 and Garage
  3900/3903 to the host; the Garage admin API has no auth. Prod compose
  publishes nothing. The delta is intentional but unguarded and undocumented.
- `docker/nginx.prod.conf:30` — HSTS commented out. Neither nginx config sets
  `X-Frame-Options`, `X-Content-Type-Options`, or CSP.
- `.env.example:84,100,205` — `POSTGRES_PASSWORD=password` plus two
  `CHANGE_THIS_TO_A_RANDOM_SECRET_KEY` placeholders. Nothing at API startup
  refuses to run on known-weak/placeholder secrets outside dev mode.

### F6 — Token handling (informational)

Web stores access+refresh tokens in `localStorage`
(`web/src/lib/auth/oauth-utils.ts`) — XSS-readable; PKCE protects the flow,
not the stored tokens. CLI stores tokens at `~/.config/kg/config.json` with
`0600` perms (correct). httpOnly-cookie refresh tokens are the eventual
upgrade path; acceptable as-is for now.

### F7 — Enforcement inventory

Exists but unwired to CI: `lint_queries.py` (soft-fail), the API auth test
suite, `docstring_coverage.py`, `lint_gating.py`, `lint_backup.py`, vulture.
Does not exist at all: secret scanning, dependency auditing
(pip-audit/npm audit/cargo-audit/dependabot), SAST, Python/TS lint in CI,
coverage reporting.

## Disposition

Decision record: ADR-401 (Deterministic Security Enforcement Baseline).
Remediation tracked as the `enforcement-baseline` issue cluster; see the
tracking issue for the live list.
